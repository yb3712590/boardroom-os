# Boardroom OS 后续 TODO（Codex 版）

> 日期：2026-04-03
> 目的：把 `CTO/` 目录里的评审、设计和时间线，结合当前仓库真实状态，收敛成一版可审阅的新 TODO 草案。
> 当前基线（2026-04-03 实测）：
> - backend：`source .venv/bin/activate && python -m pytest tests -q` -> `360 passed`
> - frontend：`npm run build` -> 成功
> - frontend：`npm run test:run` -> `20 passed`

## 先讲结论

- [ ] 接受 CTO 的核心判断：项目最大的真实缺口不是治理壳，而是 CEO 智能、真实 Worker 执行、人格模型。
- [ ] 不接受把“大规模冻结代码搬迁”作为第一步。当前主线已经是绿的，先动目录会放慢验证速度，也和现有路线纠偏里“少动外围、别大搬家”的原则冲突。
- [ ] 新主线顺序调整为：现状校准 -> 真实 Worker 主链打通 -> CEO 影子模式 -> CEO 有限接管 -> 前端拆壳 -> 人格模型 -> 会议室协议 -> 代码瘦身与后置能力隔离。
- [ ] `detailed-todo.md` 更适合作为任务库，不适合作为马上执行的唯一主线；这份文档是新的执行顺序草案。

## 和 CTO 原方案的主要差异

- [ ] CTO 方案把 CEO 放在最前面，我建议先把真实 Worker 主链跑实。原因很简单：当前仓库已经有 Provider 配置、调用路径、前端入口和测试支点，先把 live path 做成真实能力，风险更小，价值也更快可见。
- [ ] CTO 方案把前端重构抬到很早的位置，我建议把它放到智能主线稳定之后，或者只做最小拆壳并行推进。原因是当前前端的问题主要是结构膨胀，不是功能断裂。
- [ ] CTO 方案的 M0 倾向先搬运冻结代码，我建议先做“文档隔离和入口隔离”，等主线智能跑稳后，再决定是否做物理迁移。

## 校准后的现实

- [ ] 当前系统已经有可运行的本地闭环和 Web 壳，不是从 0 开始。
- [ ] `workflow_auto_advance.py` 很薄，说明后续替换成本可控，但现阶段还没有真正的 CEO 调度器。
- [ ] Runtime 已有 OpenAI Compat 配置、调用、前端设置和测试，不是“真实 Worker”完全空白。
- [ ] 前端问题主要是结构膨胀，不是功能缺失；重构应该服务后续扩展，不该先于智能主线阻塞。
- [ ] CTO 文档里有少量和当前代码不完全一致的细节，比如文件体量估计、路径写法、能力覆盖表述；执行前必须先校准。

## P0：先把主链上的智能补齐

### P0-A：现状校准与单一真相源

结果：之后任何开发都能明确知道“哪些已经真实存在，哪些只是文档目标”。

- [ ] 做一份“代码 / README / `doc/TODO.md` / CTO 文档”四方能力对照表，确认主链上每个阶段到底是真实运行、半真实、还是仅文档存在。
- [ ] 明确 Runtime 支持矩阵：按角色、输出类型、执行模式列清楚当前支持范围和缺口。
- [ ] 把冻结模块列成清单和边界说明，先在文档上隔离，不做物理搬迁。
- [ ] 修正文档里的明显漂移项，例如路径写法、过时规模估计、验证命令入口。
- [ ] 形成新的默认上下文加载顺序，让后续代理先读当前真相，而不是先读过期大纲。

验收门槛：

- [ ] 新人只看入口文档就能判断当前主线和冻结边界。
- [ ] 不再出现“文档说已经有，代码里其实还没有”的关键误判。

对应 CTO 任务：借用 `P1-CLN-*` 的边界整理思路，但不执行 `_frozen/` 迁移。

### P0-B：把真实 Worker 执行扩到当前主链

结果：现有 `project-init -> scope -> build -> check -> review -> closeout` 这条链，在配置 Provider 后能真实跑模型，而不是只靠确定性结果。

- [ ] 先把当前已接好的 OpenAI Compat 路径补成稳定能力：超时、限流、认证失败、坏响应都要有清晰分类和回退。
- [ ] 把真实执行覆盖扩到主链需要的 5 类输出：`consensus_document`、`implementation_bundle`、`delivery_check_report`、`ui_milestone_review`、`delivery_closeout_package`，以及对应的 `maker_checker_verdict`。
- [ ] 统一 live path 和 deterministic path 的结果校验、审计产物、incident 记录方式，避免两套行为分叉。
- [ ] 给 Dashboard / Incident / Review 读面补足 provider 健康和回退信号，让董事会能看见“现在到底是真实跑模型还是回退了”。
- [ ] 用 mock provider 补完整端到端测试，覆盖 build / check / review / closeout 四段。

验收门槛：

- [ ] 配好 Provider 后，一条完整 workflow 能在真实模型输出下跑到 closeout。
- [ ] Provider 出错时不会把工作流打坏，能自动降回确定性路径并留下证据。
- [ ] 后端测试仍全绿，前端构建和前端测试仍全绿。

对应 CTO 任务：优先吸收 `P0-WRK-001` 到 `P0-WRK-012`，但顺序改成“先稳定 live path，再扩输出覆盖，再补观测性”。

### P0-C：让 CEO 先进入影子模式

结果：系统开始出现真正的“CEO 判断”，但先不直接改状态，先拿真实数据比对质量。

- [ ] 定义 CEO 动作契约、状态快照、系统提示和输出校验。
- [ ] 在工单完成、失败、审批完成、incident 恢复这些节点上，让 CEO 读当前快照并产出建议动作。
- [ ] 影子模式下只记录建议、校验结果和与现有确定性调度器的差异，不执行动作。
- [ ] 为 CEO 决策加审计日志和回放测试，确保后续能解释“为什么它提这个动作”。
- [ ] 跑一轮代表性场景回放，确认 CEO 建议质量够稳定，再决定哪些动作进入有限接管。

验收门槛：

- [ ] 至少覆盖“创建首票、失败重试、升级 incident、招聘建议、等待董事会”五类场景。
- [ ] 影子模式不会影响现有主链成功率。
- [ ] 能清楚看见 CEO 建议和现有确定性推进在什么地方不一致。

对应 CTO 任务：吸收 `P0-CEO-001` 到 `P0-CEO-015`，但新增“影子模式”阶段，不直接一步切换主调度器。

## P1：在 P0 稳住后推进

### P1-A：CEO 有限接管

结果：CEO 不再只是提建议，而是开始在受控边界内真正推进工作。

- [ ] 先放开有限动作：创建工单、重试工单、升级董事会、发起招聘建议。
- [ ] 所有动作都必须经过 Reducer 级校验；校验失败自动退回确定性调度器。
- [ ] 保留现有 `workflow_auto_advance` 作为总回退，不做一次性替换。
- [ ] 明确哪些动作暂时仍留在确定性链里，例如审批后整段 follow-up 的原子写入和过于敏感的阶段推进。
- [ ] 增加“影子建议 vs 实际执行”对照读面或审计产物，方便继续校准提示词。

验收门槛：

- [ ] CEO 至少能稳定处理首票创建、失败重试、无 worker 时招聘、需要董事会时升级四类动作。
- [ ] 一旦 LLM 路径不稳，系统仍能靠现有确定性逻辑继续推进。

### P1-B：前端拆壳，只拆会继续长的部分

结果：前端不再把后续智能能力全塞回一个大文件，但也不做过度工程化。

- [ ] 先拆数据层：类型、API 客户端、SSE 管理器、主 store、review store、UI store。
- [ ] 再拆页面壳和抽屉壳：`DashboardPage`、各类 Drawer、错误边界。
- [ ] 保持视觉和交互基本不变，不在这一阶段重新设计品牌风格。
- [ ] 测试重点放在 store、API、关键交互和错误状态，不追求一次性铺满组件库。
- [ ] 把 `App.tsx` 缩到只剩路由和组装职责。

验收门槛：

- [ ] 功能不回退。
- [ ] 前端测试数量和覆盖范围明显高于当前单文件主测法。
- [ ] 后续接入 provider health、CEO 影子建议、人格信息时，不需要再往 `App.tsx` 上堆几十个状态。

对应 CTO 任务：选择性吸收 `P0-FE-001` 到 `P0-FE-022`，但不要求一上来就把所有共享组件和样式体系一次到位。

### P1-C：人格模型接入真实决策

结果：员工不再只是空字段，招聘和执行开始体现差异化。

- [ ] 定义最小可用的 `skill / personality / aesthetic` 三组维度，不追求第一版就“很像真人”。
- [ ] 先做 4 到 6 个模板，把它们接进 staffing catalog、CEO 提示词和 worker 提示词。
- [ ] 在招聘决策里加入最小多样性约束，避免全员同一模板。
- [ ] 在 Review Pack 或 Workforce 里可见当前员工画像，方便董事会判断招聘是否合理。

验收门槛：

- [ ] 同类工单换不同模板时，输出风格有可感知差异。
- [ ] CEO 招聘不再只看“有没有人”，还会看团队互补性。

## P2：功能收口与后置能力

### P2-A：会议室协议最小版

结果：遇到高耦合议题时，不用退回自由群聊，有受控的结构化协作通道。

- [ ] 只实现 1 个会议类型，建议先做 `TECHNICAL_DECISION`。
- [ ] 只支持最小轮次：立场、质疑、方案、收敛。
- [ ] 输出必须还是结构化文档，并能回到现有审查 / 审批链。

### P2-B：代码瘦身与冻结能力隔离

结果：主线阅读成本下降，但不为“整洁”破坏稳定主链。

- [ ] 先做 import 依赖图和调用清单，再决定是否物理迁移冻结模块。
- [ ] 如果迁移，放到最后做，并以“不影响主线测试”为绝对前提。
- [ ] 如果不迁移，也至少把入口、文档和目录注释补齐，让人一眼能分清主线和冻结区。

### P2-C：检索、Provider 路由、发布准备

结果：在主链智能已经成立后，再补稳定性和规模化增强。

- [ ] 检索只在真实证明本地历史摘要不够后再做，不预支向量栈。
- [ ] Provider registry / routing / fallback 策略放在真实 Worker 和 CEO 稳定之后。
- [ ] 发布准备以“本地单机演示可复现”为目标，不提前做远程控制面。

## 这版 TODO 明确不先做

- [ ] 不先做 `_frozen/` 大搬家。
- [ ] 不先做多租户、worker-admin、远程 handoff、对象存储扩张。
- [ ] 不先做向量检索或公网身份层。
- [ ] 不先做整套前端设计系统或大规模视觉翻新。
- [ ] 不把 stub、提示词样例或影子模式输出记成“能力已实现”。

## 建议首轮执行批次

如果这份新 TODO 通过，我建议第一轮就连续做下面 4 个批次：

- [ ] 批次 1：现状校准 + 支持矩阵 + 文档入口纠偏
- [ ] 批次 2：OpenAI Compat live path 稳定化 + provider 观测信号
- [ ] 批次 3：把 live Worker 覆盖扩到 build / check / review / closeout 主链
- [ ] 批次 4：CEO 影子模式 + 审计日志 + 回放测试

首轮不建议并行开的方向：

- [ ] 全量前端重构
- [ ] 人格模型
- [ ] 会议室协议
- [ ] 冻结代码迁移
