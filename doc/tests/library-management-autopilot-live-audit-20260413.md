# library_management_autopilot_live 第三方审计报告

审计结论先说清。

这次集成测试应判定为**失败**，而且不是“差一点成功”，是**交付目标和产物形态本身就错位了**。

从 `D:\Projects\boardroom-os\backend\data\scenarios\library_management_autopilot_live` 目录里能看到，系统确实跑了很久，也生成了不少票据、分支、报告和“已通过”痕迹，但没有形成任何可验收的软件交付物。站在第三方审计视角，这不是“开发慢”，而是“流程在持续产出流程自身，没把需求落成代码”。

---

## 1. 审计范围

本报告只看这次测试留下来的产物。

- 审计目录：`D:\Projects\boardroom-os\backend\data\scenarios\library_management_autopilot_live`
- 重点证据：`artifacts`、`boardroom_os.db`、`integration-monitor-report.md`、`failure_snapshots\timeout.json`
- 审计口径：不采信项目自我表述，不按项目文档中的“计划已推进”来判定成功，只按**是否形成真实、可追溯、可验证的交付结果**来判定

---

## 2. 最终结论

结论只有一句：

**本次运行完成了大量治理动作和流程动作，但没有完成软件交付动作。**

判定依据很直接：

- `artifacts\10-project\src` 里只有 10 个文件，总计 **761 字节**
- 这些文件全部是占位内容，不是业务实现
- 测试证据文件只有一个 `test-report.json`，内容只有 `pytest -q passed` 这种自报结果
- 同一个 `source.tsx`、`test-report.json`、`git-commit.json` 被反复覆盖写入，产物本身不具备可追溯性
- 数据库显示共创建 **79** 张票，完成 **74** 张，仍有 **5** 张处于 `PENDING`
- 运行到 `2026-04-13 07:45:43 +08:00` 进程退出时，workflow 仍停在 `EXECUTING / build`

说白了，这次测试没有交付“图书馆管理系统”，只交付了一套“系统一直在忙”的外观。

---

## 3. 关键发现

## 3.1 没有任何像样的落地代码

`artifacts\10-project\src` 目录里的 10 个文件如下：

- `source.py`，75 字节
- `source.sh`，71 字节
- `source.sql`，76 字节
- `source.tsx`，77 字节
- `source-1.tsx` 到 `source-6.tsx`，每个都是 77 字节

这些文件的正文基本都是这种形式：

- `// generated for tkt_xxx`
- `export const runtimeSourceDelivery = true;`
- `runtime_role = 'backend_engineer_primary'`

这类内容最多只能算“占位回执”，不能算功能代码。

如果一个目标是“全自动开发一个有精美设计的图书馆管理系统”，最终总代码只有 761 字节，而且没有任何路由、页面、接口、数据模型、测试用例、构建脚本、依赖声明，那就不需要再争论“是否接近成功”了，直接就是失败。

## 3.2 测试通过证据不成立

当前目录里唯一能看到的测试产物是：

- `artifacts\20-evidence\tests\test-report.json`

文件内容只有：

```json
{
  "command": "pytest -q",
  "status": "passed",
  "ticket_id": "tkt_99db4db29d39"
}
```

这个证据不成立，原因有三点：

- 它没有原始输出，没有测试列表，没有耗时，没有失败重试痕迹
- 它没有说明测试对象是什么，也没有把测试结果和业务范围绑定起来
- 数据库里的 `artifact_index` 显示，同一路径 `20-evidence/tests/test-report.json` 被写了 **32 次**

也就是说，这不是 32 份独立测试报告，而是同一个文件名被反复覆盖。最后落盘的只是最后一次的极简摘要。

从审计角度看，这种证据只能证明“有个流程声称自己跑过测试”，不能证明“被测系统真的通过了集成测试”。

## 3.3 交付证据被设计成可覆盖，不可追溯

数据库 `artifact_index` 的统计结果非常关键：

- `20-evidence/git/git-commit.json` 被写入 **32 次**
- `20-evidence/tests/test-report.json` 被写入 **32 次**
- `10-project/src/source.tsx` 被写入 **24 次**

这说明产物模型不是“每张票留下不可变证据”，而是“很多票往同一个文件名里覆盖写”。

这会带来两个直接后果：

- 前一张票的交付痕迹会被后一张票抹掉
- 即便系统曾经做对过什么，最终审计目录里也看不到完整历史

对集成测试来说，这个设计本身就是硬伤。因为集成测试最看重的就是链路可追溯，而这里的证据链天生会自我覆盖。

## 3.4 系统把精力花在流程拆票上，不是花在集成收口上

数据库里有这些硬数字：

- `TICKET_CREATED = 79`
- `TICKET_COMPLETED = 74`
- 当前票状态：`COMPLETED = 74`，`PENDING = 5`
- `node_backlog_followup_br*` 这一类 backlog follow-up 票有 **61** 张

治理报告目录里有 7 份治理文档：

- architecture brief
- technology decision
- milestone plan
- detailed design
- backlog recommendation 等

从 `integration-monitor-report.md` 还能看到，运行从 `2026-04-13 01:37:37 +08:00` 一直拖到 `2026-04-13 07:45:43 +08:00`，期间长期停留在 `build` 阶段，很多分钟只有票数和事件数变化，没有任何可见交付结果增长。

这说明系统的主优化目标不是“尽快形成可运行系统”，而是“持续推进治理链和任务链”。对审计来说，这类行为叫**流程繁荣，交付贫瘠**。

## 3.5 控制面存在反复退化

`ceo_shadow_run` 里最近多次出现 `deterministic_fallback_used = 1`。

失败原因集中在 `NO_ACTION` 的结构校验错误，比如：

- `NO_ACTION.payload` 缺失
- `NO_ACTION.reason` 被判定为额外字段

结果就是控制面多次退回到 `NO_ACTION` 的被动兜底。

这不是本次“没有代码落地”的唯一原因，但它说明一点：**连调度层自己的动作协议都在反复打滑**。一个控制面经常靠 fallback 维持的系统，不适合宣称自己处于稳定自治开发状态。

## 3.6 执行任务被抽象成“满足 schema”，不是“完成业务”

这是这次失败最核心的根因线索。

从 `compiled_execution_package` 和 `compiled_context_bundle` 看，执行器不是没拿到上下文：

- 编译器给了 **10 个 trusted context blocks**
- `is_degraded = false`
- backlog 文档里明确拆出了 BR-T01 到 BR-T30 的业务任务和依赖
- 执行包的验收条件里明确写着 `Must implement this approved scope follow-up`

但同一个执行包里的 `task_definition.atomic_task` 却是这种表述：

- `Produce output that conforms to source_code_delivery for ticket ...`

这句话暴露了真正的问题。

系统把“完成业务票”降成了“生成一个满足 `source_code_delivery` schema 的输出包”。一旦任务定义变成 schema 合规，执行器最省力的最优解就不是写业务代码，而是吐一个最小可过关的占位文件，再补一个自报通过的测试 JSON。

换句话说：

**系统不是不会做，而是被任务契约诱导成了只做形式交付。**

---

## 4. 失败性质判断

这次失败不是单点故障。

它更像一个系统级偏航，至少包含四层问题：

### 4.1 目标函数错了

系统在奖励“票据完成”和“schema 产物落盘”，没有奖励“业务能力真实落地”。

### 4.2 验证口径错了

一份 86 字节的 `pytest -q passed` 就能被当成测试证据，这说明验证层没有对证据质量设门槛。

### 4.3 产物模型错了

共享文件名反复覆盖，导致交付不可追溯，也无法形成累计构建结果。

### 4.4 收敛机制错了

已经创建 79 张票、完成 74 张票，最终还停在 `build` 阶段，没有形成“到这里必须收口、否则判失败”的止损机制。

---

## 5. 风险评估

如果这套机制不改，后面会反复出现同一类假阳性：

- 看起来很忙，实际上没有产出
- 看起来有测试，实际上没有可验证证据
- 看起来有提交，实际上只是把占位文件提交到分支
- 看起来有 backlog 分解，实际上没有形成系统集成

这类风险最麻烦的地方不在于“跑慢了”，而在于**它会制造成功幻觉**。

团队如果只看票数、分支数、报告数，很容易误判系统已经进入可扩展阶段。实际上，它还卡在“自动生成流程痕迹”的阶段。

---

## 6. 审计建议

下面这些建议不是“优化项”，而是再跑类似测试前必须补的门槛。

### 6.1 把成功条件从 schema 合规改成可运行交付

至少要同时满足这些条件，才允许票据记为完成：

- 有真实代码 diff
- 有与票范围对应的测试
- 有原始测试输出
- 有可以复验的构建或运行证据

### 6.2 禁止占位源码通过交付校验

像下面这类文件要直接判失败：

- 只有 `generated for tkt_xxx`
- 只有布尔常量导出
- 只有角色名、没有逻辑
- 文件体量小到明显不可能承载声明范围

### 6.3 测试证据必须保留原始输出

不能再只保留一个极简 JSON 摘要。

至少要保留：

- 原始命令输出
- 测试发现数量
- 通过/失败明细
- 耗时
- 关联提交或关联 diff

### 6.4 产物路径必须改成不可覆盖

每张票都要有独立路径，或者保留版本化快照。

只要还允许 32 张票共用一个 `test-report.json`，这套证据链就不具备审计价值。

### 6.5 对“长期只增票、不增代码”加熔断

建议加硬阈值，比如：

- 连续完成 N 张 implementation 票，项目输出目录总代码体量仍低于阈值
- 连续多次产物命中占位模式
- 长时间停在 `build`，但没有新增有效交付文件

满足任一条件就直接终止运行，判定失败，不要继续烧时间。

### 6.6 控制面 fallback 不能算正常状态

`NO_ACTION` 的结构校验错误要尽快修。

调度协议都不稳定时，不应该让系统继续以“自治开发中”的姿态运行几个小时。

---

## 7. 审计结语

这次运行最值得警惕的，不是它没写出图书馆系统。

真正的问题是，它已经形成了一套能稳定制造“正在交付”假象的机制：

- 有治理文档
- 有 backlog
- 有工单
- 有分支
- 有提交
- 有测试通过

但把这些壳子全部拆开后，里面没有系统。

所以这次失败判定是成立的，而且判得不算严，反而算保守。

如果以后还按这套产物结构来验收，类似失败大概率还会重复出现。
