## 二、逐条分析

- **建议**：无需额外排期，保留在 M6 内按原计划推进。实施时确保 `preferred_model` / `actual_model` 写入事件流，以满足审计需求。

#### #61 CTO / 架构师默认产物为治理文档

#### #62 CEO / 文档角色 / 实施 Worker 模型解耦

- **影响等级**：🟢 低，已在规划内。

### 2.2 可作为现有里程碑自然增强的条目

#### #63 Worker 应知晓最小组织存在

- **当前状态**：当前执行包 (`compiled_execution_package`) 携带 `persona_summary`，但不含上下游协作者信息。
- **里程碑归属**：建议挂靠 M2（最小可行 Worker）或 M7（集成打磨）。
- **实施方式**：在执行包中新增 `org_context` 字段，至少包含 `upstream_provider`、`downstream_reviewer`、`escalation_path`。数据来源是 workflow 投影 + 员工投影，不需要新的持久化。
- **建议**：作为 M2 的小增强纳入，不单独成为里程碑。
- **影响等级**：🟢 低，执行包字段扩展。

#### #64 REQUIREMENT_ELICITATION 协议

- **当前状态**：CEO 当前在 `project-init` 后直接发起 scope kickoff 票。如果董事会指令信息不足，当前没有受控的补充澄清机制。
- **里程碑归属**：建议在 M1（最小可行 CEO）之后、M7 之前安排。
- **实施方式**：
  1. 新增 `BOARD_REVIEW_REQUIRED` 子类型 `REQUIREMENT_ELICITATION`
  2. CEO snapshot 评估输入完整度，低于阈值时触发
  3. 产出结构化选择题/确认项模板
  4. 董事会回复后 CEO 继续正常拆解
- **建议**：作为独立小任务挂到 M7 前，约 2–3 天工量。当前 CEO 已支持 `ESCALATE_TO_BOARD`，扩展难度不大。
- **影响等级**：🟡 中。改变了项目初始化的入口逻辑，需要确保不影响现有 `project-init` 闭环。

#### #65 SPIKE_TICKET 探索型工单

- **当前状态**：当前 ticket 类型不区分"探索"和"实施"。所有票都走同一条 `maker → checker → fix/incident` 流水线。
- **里程碑归属**：建议放在 M2 之后，M7 之前。
- **实施方式**：
  1. 新增 `ticket_type = SPIKE`，与标准工单共用生命周期事件
  2. Spike 产物标记为 `non_mainline`，不触发主线合入
  3. 合法输出 schema 限定为 `adr@1` / `exploration_report@1`
  4. CEO 在 spike 完成后，基于产物重新创建标准工单
- **建议**：作为 M7 的前置增强，约 3–4 天工量。注意隔离语义不能过重——MVP 阶段不需要真正的沙箱，只需标记 + 产物路径分离。
- **影响等级**：🟡 中。引入新 ticket 类型，需要调度器、投影和前端都识别。

#### #66 CEO "业务认可、技术质疑" + 复用优先

- **当前状态**：CEO 当前按 snapshot 直接产出动作，没有"先评估复用"这层逻辑。
- **实施方式**：主要是 CEO prompt 和 `ceo_prompts.py` 的行为调整，加上 snapshot 中暴露"已知可复用组件"索引。
- **建议**：作为 CEO prompt 增强纳入 M7。不需要新的后端模块，只需 prompt 策略 + 可选的 reuse registry 投影。
- **影响等级**：🟢 低，prompt 层变更。

#### #67 并发实施、串行收口策略

- **当前状态**：当前 workflow 是严格串行的 `scope → BUILD → CHECK → REVIEW → closeout`。
- **实施方式**：这条更像治理策略宣示，而不是功能需求。当前串行主线已经在运行，并发实施要等 CEO 能同时管理多 ticket 并行后才有意义。
- **建议**：记录为治理原则，不单独排入开发。等 CEO 成熟到能管理并发 ticket 流后，再按需实施并行调度。
- **影响等级**：🟢 低，当前阶段仅作原则记录。

#### #68 ADR / 决策文档自动产出

- **当前状态**：会议室当前产出 `consensus_document@1`，但没有明确的 ADR 格式。
- **里程碑归属**：建议挂靠 M5（会议室协议）增强，或 M7 收口时补。
- **实施方式**：
  1. 扩展 `consensus_document@1` schema，增加 ADR 标准字段
  2. 会议共识文档自动标记为 `decision_record`
  3. 旧讨论材料标记 `archived`，执行包默认只引用 ADR
- **建议**：和 M5 一起推进，约 1–2 天增量。当前会议室已有结构化共识产出，只需 schema 扩展。
- **影响等级**：🟢 低，schema 增强。

#### #75 "实施完成"定义为代码+配置+测试+证据+文档原子闭环

- **当前状态**：当前 `TICKET_COMPLETED` 只做 schema validation 和 write-set validation，不检查文档同步。
- **实施方式**：这条本质是治理规则的强化。MVP 阶段可以先在 closeout checker 的审查标准中加入"文档是否同步"维度，而不是改 ticket 状态机。
- **建议**：作为 M7 的 checker 增强纳入。前期通过 checker prompt 实现软约束，后期再考虑硬性 gate。
- **影响等级**：🟡 中。如果做成硬性 gate 会影响整条流水线的完成定义；建议 MVP 阶段先做软约束。

### 2.3 需要新增独立能力或子系统的条目

#### #69 分层上下文架构（L1/L2/L3）⚠️ 重大影响

- **当前状态**：Context Compiler 当前已能处理文本、媒体、下载型附件和本地历史摘要，产出可审计执行包。但上下文组装是"平铺式"的，没有显式的 L1/L2/L3 分层。
- **影响分析**：
  - **L1（执行工作区）**：当前执行包已经接近 L1 的定义，只需收紧字段边界。影响小。
  - **L2（按需视图层）**：需要 Context Compiler 集成 RAG / FTS5 / LSP 等检索手段。这依赖 M6 的 FTS5 先落地，且需要定义 `context_query_plan` 的执行协议。影响中。
  - **L3（持久存储层）**：当前 SQLite 事件流 + artifact 存储已经是 L3，但缺少"Agent 通过受控工具显式请求"的提升机制。需要定义工具调用协议。影响大。
- **建议**：分三步走——
  1. **M2 阶段**：收紧执行包为严格 L1（已接近完成）
  2. **M6 阶段**：FTS5 落地后，实现 L2 的按需检索注入
  3. **M7 之后新增 M6.5**：实现 L3 → L2 的受控提升协议
- **影响等级**：🔴 高。这是对 Context Compiler 核心架构的重新定义，需要跨多个里程碑渐进实施。

#### #70 项目地图（Project Map）⚠️ 重大影响

- **当前状态**：当前没有结构化的项目地图。Context Compiler 基于 ticket 引用和 artifact 索引工作，不具备模块边界、接口依赖、责任归属等结构化认知。
- **影响分析**：
  - 项目地图本质是一个**持续更新的结构化知识图谱**，需要：代码分析（模块边界、接口依赖）、事件流分析（责任归属、失败热点）、产物血缘追踪
  - 这不是一个可以在几天内完成的功能，而是一个需要持续维护的子系统
  - 如果做得太重，会违反当前"本地单机 MVP"的阶段约束
- **建议**：MVP 阶段降级为**最小项目地图**——
  1. 手工维护一份 `project_map.json`，描述模块边界和接口依赖
  2. Context Compiler 在编译时读取该文件，用于判断 ticket 的影响范围
  3. 后续再考虑自动化更新（代码分析、事件流挖掘）
- **影响等级**：🔴 高。完整实现是一个独立子系统；MVP 阶段建议只做静态最小版。

#### #71 标准化 Evidence Pack ⚠️ 重大影响

- **当前状态**：当前 maker-checker 产出结构化裁决，closeout 产出 `delivery_closeout_package@1`，但没有统一的"证据包"概念。各阶段的证据散落在不同 artifact 中。
- **影响分析**：
  - 统一 Evidence Pack 意味着每个可交付产物都必须附带标准化的证据集合
  - 这会改变 `TICKET_COMPLETED` 的验收逻辑——不只检查 schema 和 write-set，还要检查 evidence pack 完整性
  - 影响范围：ticket 状态机、checker 审查标准、closeout 流程、前端 Review Room 展示
- **建议**：分两步走——
  1. **M7 阶段**：定义 `EvidencePack` schema，在 closeout 和 final review 中要求附带（软约束）
  2. **M8 之后**：逐步推广到所有 maker-checker 节点（硬约束）
- **影响等级**：🔴 高。如果一步到位做硬约束，会显著增加每个 ticket 的完成成本。

#### #72 组织过程资产体系

- **当前状态**：当前有 `memory-log.md`、`context-baseline.md` 和归档机制，但这是人工维护的文档，不是系统化的过程资产。
- **影响分析**：
  - 完整的过程资产体系需要：自动归档、摘要生成、可追溯引用、审计线索保留
  - 这依赖 #69 的 L3 存储层和 #70 的项目地图
  - MVP 阶段不适合做完整实现
- **建议**：当前阶段只做两件事——
  1. 确保所有 ADR、审查结论、证据包都写入 artifact 存储并建立索引
  2. Context Compiler 在编译时能检索到相关历史决策
  - 完整的过程资产管理放到 M8 之后
- **影响等级**：🟡 中。MVP 阶段降级为"artifact 索引 + 检索"即可。

#### #73 组织学习闭环 ⚠️ 重大影响

- **当前状态**：当前有 incident 和熔断机制，但失败经验没有被结构化提炼和复用。
- **影响分析**：
  - 完整的组织学习闭环需要：失败指纹提取、经验标注、适用范围验证、策略反馈
  - 这是一个**元治理能力**——系统不只是执行任务，还要从执行中学习并修正自身
  - 实现复杂度高，且效果难以在 MVP 阶段验证
- **建议**：MVP 阶段只做最小闭环——
  1. incident 复盘时自动生成 `lesson_learned@1` artifact
  2. CEO snapshot 在调度时能检索到相关历史 lesson
  3. 完整的"经验 → 策略修正"闭环放到 M8 之后
- **影响等级**：🔴 高。完整实现是系统级能力变更；MVP 阶段建议只做 artifact 层面的最小闭环。

#### #74 文档治理体系

- **当前状态**：当前项目文档体系已经比较成熟（`mainline-truth.md`、`roadmap-reset.md`、`TODO.md` 等分层），但这是人工维护的，不是系统强制的。
- **影响分析**：
  - 作为框架能力，需要系统能定义文档分类、层级、时效性和治理责任
  - 需要和 #75 的"实施完成定义"联动
  - MVP 阶段不需要做成系统功能，当前人工文档体系已经够用
- **建议**：当前阶段只做两件事——
  1. 在 `feature-spec.md` 中记录为长期治理原则
  2. 在 checker 审查标准中加入"文档是否按规范更新"维度
  - 系统化的文档治理引擎放到 M8 之后
- **影响等级**：🟡 中。MVP 阶段作为治理原则记录，不做系统实现。

## 三、重大影响条目汇总

以下 4 条如果按原文完整实现，会对当前架构和开发节奏产生显著影响：

| 条目 | 核心影响 | MVP 建议 | 完整实现时机 |
|------|----------|----------|--------------|
| #69 分层上下文 | 重新定义 Context Compiler 架构 | 收紧 L1 + M6 落地 L2 | M8 之后 |
| #70 项目地图 | 引入持续维护的知识图谱子系统 | 手工静态 `project_map.json` | M8 之后 |
| #71 Evidence Pack | 改变所有产物的完成定义和验收逻辑 | M7 在 closeout 软约束 | M8 之后 |
| #73 组织学习闭环 | 引入元治理能力，系统从执行中学习 | incident 复盘产出 lesson artifact | M8 之后 |

**核心建议**：这 4 条都应该在 MVP 阶段做**最小降级版**，把完整实现推到 M8 之后的"平台增强"阶段。原因：

1. 当前主线闭环已经可跑，这些能力不直接缩短 MVP 路径
2. 完整实现需要跨多个模块的协调变更，风险高
3. MVP 阶段的降级版已经能提供基本价值
