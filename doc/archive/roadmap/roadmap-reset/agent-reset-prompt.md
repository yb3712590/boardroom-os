# Boardroom OS 开发代理重置提示词

> 这份文档从 `doc/roadmap-reset.md` 拆出，保留给需要整段复制提示词的场景。当前规则解释请先看 `../roadmap-reset.md` 与 `rationale.md`。

```md
基于当前 Boardroom OS 仓库继续开发，只做增量推进，不要重建骨架，不要把仓库当成空白项目。

先建立上下文，顺序固定：
1. 读根 `README.md`
2. 读 `doc/README.md`
3. 读 `doc/mainline-truth.md`
4. 读 `doc/roadmap-reset.md`
5. 读 `doc/TODO.md`
6. 读 `doc/history/context-baseline.md`
7. 读 `doc/history/memory-log.md`
8. 只按需读与本轮方向直接相关的设计文档；不要通读全部设计文档
9. 只有在需要精确历史原因、原始验证记录或旧路径兼容细节时，才去看 `doc/history/archive/*`

先用这些文档建立“当前主线”判断，不要被仓库里已经存在、但已降级为后置能力的实现带偏。

推进基线：
- 当前唯一主线是“本地单机 Agent Delivery OS MVP”
- 事件流与投影是真相源，Ticket 驱动无状态执行器推进工作
- `Inbox -> Review Room` 和 Maker-Checker 是真实治理闭环，不是装饰页面
- React Boardroom UI 只是最薄治理壳，前端不拥有工作流真相
- 已存在的 `auth / worker-admin / 多租户 scope / 对象存储 / 远程 handoff` 默认视为保留但冻结的外围能力
- `Search / Retrieval`、公网身份层、远程多节点控制面默认不是当前优先方向，除非我明确要求

本轮默认采用“单一方向下的连续切片批次”：
- 当前默认只从这些方向里选一个：`Runtime / Backend`、`Workflow Governance`、`Context Compiler`、`Frontend / Boardroom UI`、`Packaging / Dev Experience（仅限直接缩短本地 MVP 路径时）`
- 一次规划 `2` 到 `4` 个前后衔接、强相关的连续切片
- 这些切片必须属于同一条功能链路，共享主要文件、契约、状态机或验证链
- 如果后一个切片明显依赖前一个切片，且合并实施风险可控，默认这一轮一并做完
- 规划后默认直接连续实施，不要做完第一片就停下来等确认
- 不要同时跨多个无关方向，也不要为了做多而越过本轮边界

强约束：
- 不要为了“顺手优化”去扩张范围；发现支线问题时，先判断它是否直接缩短 `Board -> Review` 路径，不能就记录到文档后停止扩张
- 主线外代码只允许做最小解堵，不允许顺势继续扩接口、补抽象、重写目录或平台化
- 不要围着单点内部细节钻牛角尖；只要当前链路可运行、可验证、文档真实，就继续往下一个连续切片推进
- 不改无关模块，不做大面积重命名，不为未来假想需求提前铺扩展点
- 如果设计与代码不一致，先说明差异，再做最保守处理；不要借机重写已工作的链路
- 不要把未实现能力写成已经完成，不要复述过期结构，不要用旧文档覆盖当前代码现实

选择方向时优先：
1. 当前代码已有明显支点、继续往前推最顺手的方向
2. `doc/TODO.md` 里仍未完成、且 `memory-log` 最近明确提到过的方向
3. 能在一轮内形成“入口 -> 实现 -> 验证 -> 文档同步”闭环的方向
4. 不优先新开空白大方向；后置方向除非我明确要求，否则不选

如果眼前有多个真实可走的方向，先用很短篇幅比较候选方向。每个候选都必须说明：
1. 现实场景
2. 这轮做完会得到什么
3. 好处
4. 代价/风险
5. 不选会怎样

如果默认路径已经很明显，也要先简短说明另一个可选路径为什么这轮不选，然后明确继续默认路径。

开始前必须先输出：
1. 当前代码状态总结
2. 本轮选择的单一方向
3. 本轮准备连续完成的 `2` 到 `4` 个切片
4. 这些切片分别对应的现实场景和做完后的直接结果
5. 本轮明确不做的内容
6. 然后直接开始实现，不要停在计划阶段等确认

表达要求：
- 全程用通俗、自然、好懂的中文
- 先讲结论，再讲细节
- 少用英文术语，必要时顺手解释
- 不要写得像咨询报告
- 制定计划时，必须把“这次改代码对应的现实场景”讲清楚，不要只堆开发术语
- 尽量说成人能代入的结果，例如“这轮做完后，排障时不再只能翻本地命令，而是能直接从统一读面看到 worker 状态”

实现要求：
- 必须交付可运行代码
- 必须补必要的最小测试或最小验证
- 必须同步必要文档
- 如果环境缺依赖导致某些测试或命令无法运行，要明确说明缺什么、卡在哪里，并补充你实际完成的其他验证
- 同一条功能链路上的改动默认这一轮一并完成，不要再拆碎

文档要求：
- 若本轮影响项目对外状态、已实现能力、运行方式、已知限制，更新 `README.md`
- 若本轮改变路线、关闭待办、拆分待办或调整优先级，更新 `doc/TODO.md`
- 若本轮改变文档入口、分工或索引关系，再更新 `doc/README.md`
- 每轮结束都要追加更新 `doc/history/memory-log.md`
- 所有文档都必须反映真实状态，不要预写未来能力
- 如果本轮未影响 `README.md`、`doc/TODO.md` 或 `doc/README.md`，必须明确说明为什么没改

完成后按这个顺序汇报：
1. 改了哪些文件
2. 落地了哪些能力
3. 哪些仍是 mock / stub / placeholder
4. 做了哪些测试或验证，哪些因环境限制没法做
5. 更新了哪些文档
6. `README.md` 是否更新；如果没更新，说明原因
7. `doc/TODO.md` 是否更新；如果没更新，说明原因
8. 下一步在同一方向下最合理的连续切片是什么

本轮目标：
<用一句能落到现实场景的话写清楚这轮目标，不要只写模块名>
```
