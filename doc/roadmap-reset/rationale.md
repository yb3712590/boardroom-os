# Boardroom OS 路线纠偏决议

> 生效时间：2026-03-31

## 1. 这份决议解决什么问题

项目最近几轮实现明显偏向了鉴权、worker 运维控制面、多租户 scope、对象存储和远程交付链路。它们本身不是错误方向，但已经开始挤压项目原始目标：

- 本地运行
- 事件溯源状态总线
- Ticket 驱动无状态 Agent Team
- 真实的审查与审批闭环
- 最后套一层轻量 Web 可视化壳

这份决议的目的，是把主线重新拉回“本地单机 Agent Delivery OS MVP”。

## 2. 重新确认的产品定义

Boardroom OS 当前阶段不是：

- 公网多租户 SaaS
- 远程 worker 运维平台
- 云原生基础设施项目
- 以鉴权和接入边界为中心的控制面

Boardroom OS 当前阶段应当是：

- 一个本地优先、自托管优先的 Agent Delivery OS 原型
- 一个用结构化事件、结构化状态和结构化工单治理 Agent Team 的系统
- 一个先把执行闭环和治理闭环做短、做稳，再补前端壳的产品

## 3. 当前阶段唯一主线

从现在起，项目主线固定为：

1. 本地单机可运行
2. 事件流与投影是系统真相源
3. Ticket 驱动无状态执行器推进工作
4. Maker-Checker 与 Review Room 构成真实质量闭环
5. React Boardroom UI 只做最薄治理控制壳

任何不直接服务这条主线的新增复杂度，都默认延后。

## 4. MVP 的完成定义

只有当下面几件事都成立，才算当前阶段 MVP 真正完成：

- 用户可以在本地启动系统并初始化一个 workflow
- CEO / Worker 路径可以把任务拆分并推进到可交付结果
- 执行过程由 Ticket、事件和投影驱动，而不是长对话记忆驱动
- Maker-Checker review loop 真正落地，而不是只停留在文档层
- Board 审批可以通过 `Inbox -> Review Room` 查看和裁决
- 有一个最薄的 Web UI 能看 `dashboard / inbox / review room`

当前进度补记：

- `Inbox -> Review Room` 这条董事会审批链已经真实可用
- 视觉里程碑已经具备最小 `Maker -> Checker -> Review Room` 闭环
- 但 Maker-Checker 目前仍只覆盖 `VISUAL_MILESTONE + ui_milestone_review@1`，所以整体里程碑还不能算完全关闭

## 5. 已实现但降级为后置能力

下面这些能力已经在仓库中存在，但从这份决议开始降级为“保留、不扩张、非主线”：

- 多租户 worker scope 绑定
- `worker-admin` HTTP 运维控制面
- 操作人签名令牌、可信代理断言、独立 auth rejection 读面
- multipart artifact upload
- 本地默认 + 可选对象存储双后端
- 更偏远程 worker handoff 的交付链路

处理原则：

- 不主动删除现有实现
- 不把它们继续包装成当前阶段核心卖点
- 不继续为它们新增大块设计和实现预算
- 只有在它们直接服务本地 MVP 时，才允许做最小必要修改

## 6. 当前阶段明确后置的方向

在本地 MVP 收口完成前，下面这些方向默认冻结：

- 新的 `worker-admin` 读写接口
- 更细粒度的租户隔离与租户级运维能力
- 面向公网的身份层、外网暴露策略和强信任边界
- 浏览器直传、云厂商预签名直传、远程对象存储链路增强
- 为远程多节点运行设计的新复杂控制面

## 7. 当前阶段必须推进的能力

接下来优先级最高的能力是：

- employee hire / replace / freeze 生命周期
- 把只覆盖视觉里程碑的 Maker-Checker review loop 扩到更多关键产物
- 能稳定支撑本地执行的 Context Compiler
- 最薄的 React Boardroom UI

这些能力比继续补 `auth / worker-admin / object store` 更符合项目起点，也更能尽快形成可体验产品。

## 8. 对现有代码的处理原则

这次纠偏不是一次“清库重来”，而是一次“主线收口”。

因此：

- 保留已有 backend 基础和可运行切片
- 允许继续使用现有 projection、artifact、runtime、worker-runtime 读面
- 对已有重型基础设施代码采取“冻结、收口、少动”的策略
- 新开发优先补主链路缺口，而不是再给外围控制面加层次

## 9. 后续开发的判断规则

以后每个新任务在进入实现前，都先回答一个问题：

“它是否直接缩短了本地单机 MVP 从 Board 指令到 Review 交付的路径？”

如果答案是：

- `是`：可以进入当前主线
- `不是，但能明显降低主线实现成本`：按辅线评估
- `不是`：默认后置

## 10. 文档与叙事约束

从这份决议开始：

- `README.md` 必须把“本地无状态 agent team + 最薄 web 壳”写成第一叙事
- `doc/TODO.md` 只保留当前主线待办
- 历史文档仍保留真实演进记录，但不再代表当前阶段优先级

这不是否认已经做过的工作，而是明确项目现在真正要冲向哪里。

## 11. 防止路线跑偏的执行强约束

为了避免开发时钻进局部细节、把一轮工作膨胀成外围平台建设，后续推进默认再加下面这些硬约束：

- 每轮只允许选一个方向推进，不要同时跨多个无关方向。
- 一次默认做 `2` 到 `4` 个前后衔接、强相关的连续切片；这些切片必须共享主要文件、契约、状态机或验证链。
- 如果后一个切片明显依赖前一个切片，且合并实施风险可控，默认这一轮直接连续做完，不要做完第一片就停下来等下一轮。
- 主线外代码只允许做“最小解堵”。如果为了打通当前闭环，不得不碰到 `auth`、`worker-admin`、多租户 scope、对象存储、远程 handoff 这类已降级能力，只能做最小必要修改，不能顺势继续扩接口、补抽象、加新控制面。
- 不要因为“顺手可以一起补”就扩张范围。发现支线问题时，先回到判断题：它是否直接缩短了 `Board -> Review` 的本地交付路径？如果不能，就记录后停止扩张。
- 不要围着单点内部细节钻牛角尖。只要当前链路已经可运行、可验证、文档真实，就继续往下一个连续切片推进，不为架构纯度、抽象美观或未来想象补额外复杂度。
- 不做大面积重命名、不重排目录、不重建骨架、不为未来假想需求提前铺扩展点。
- 阅读范围也要收口：除固定入口文档外，只按需读取与本轮方向直接相关的设计文档；历史 archive 只在需要精确原因、原始验证记录或旧兼容细节时才打开。
- 如果设计与代码不一致，先说明差异，再做最保守、最接近现状、最能保住主线速度的处理；不要借机重写已经工作的链路。
- 任何汇报和文档都只能陈述已实现、已验证、已暴露的真实状态，不能把计划、stub 或 placeholder 写成已经完成。

## 12. 推荐给开发代理的重置提示词

下面这版提示词与当前仓库文档同步，重点是把“先读纠偏决议、单一方向连续切片、主线外只允许最小解堵”固定下来，避免再次滑回外围基础设施扩张：

```md
基于当前 Boardroom OS 仓库继续开发，只做增量推进，不要重建骨架，不要把仓库当成空白项目。

先建立上下文，顺序固定：
1. 读根 `README.md`
2. 读 `doc/README.md`
3. 读 `doc/roadmap-reset.md`
4. 读 `doc/TODO.md`
5. 读 `doc/history/memory-log.md`
6. 只按需读与本轮方向直接相关的设计文档；不要通读全部设计文档
7. 只有在需要精确历史原因、原始验证记录或旧路径兼容细节时，才去看 `doc/history/archive/*`

先用这些文档建立“当前主线”判断，不要被仓库里已经存在、但已降级为后置能力的实现带偏。

推进基线：
- 当前唯一主线是“本地单机 Agent Delivery OS MVP”
- 事件流与投影是真相源，Ticket 驱动无状态执行器推进工作
- `Inbox -> Review Room` 和 Maker-Checker 是真实治理闭环，不是装饰页面
- React Boardroom UI 只是最薄治理壳，前端不拥有工作流真相
- 已存在的 `auth / worker-admin / 多租户 scope / 对象存储 / 远程 handoff` 默认视为保留但冻结的外围能力
- `Search / Retrieval`、公网身份层、远程多节点控制面默认不是当前优先方向，除非我明确要求

本轮默认采用“单一方向下的连续切片批次”：
- 当前默认只从这些方向里选一个：`Runtime / Backend`、`Workflow Governance`、`Context Compiler`、`Frontend / Boardroom UI`、`Packaging / Dev Experience（仅限直接缩短本地 MVP 路径时）`
- 一次规划 `2` 到 `4` 个前后衔接、强相关的连续切片
- 这些切片必须属于同一条功能链路，共享主要文件、契约、状态机或验证链
- 如果后一个切片明显依赖前一个切片，且合并实施风险可控，默认这一轮一并做完
- 规划后默认直接连续实施，不要做完第一片就停下来等确认
- 不要同时跨多个无关方向，也不要为了做多而越过本轮边界

强约束：
- 不要为了“顺手优化”去扩张范围；发现支线问题时，先判断它是否直接缩短 `Board -> Review` 路径，不能就记录到文档后停止扩张
- 主线外代码只允许做最小解堵，不允许顺势继续扩接口、补抽象、重写目录或平台化
- 不要围着单点内部细节钻牛角尖；只要当前链路可运行、可验证、文档真实，就继续往下一个连续切片推进
- 不改无关模块，不做大面积重命名，不为未来假想需求提前铺扩展点
- 如果设计与代码不一致，先说明差异，再做最保守处理；不要借机重写已工作的链路
- 不要把未实现能力写成已经完成，不要复述过期结构，不要用旧文档覆盖当前代码现实

选择方向时优先：
1. 当前代码已有明显支点、继续往前推最顺手的方向
2. `doc/TODO.md` 里仍未完成、且 `memory-log` 最近明确提到过的方向
3. 能在一轮内形成“入口 -> 实现 -> 验证 -> 文档同步”闭环的方向
4. 不优先新开空白大方向；后置方向除非我明确要求，否则不选

如果眼前有多个真实可走的方向，先用很短篇幅比较候选方向。每个候选都必须说明：
1. 现实场景
2. 这轮做完会得到什么
3. 好处
4. 代价/风险
5. 不选会怎样

如果默认路径已经很明显，也要先简短说明另一个可选路径为什么这轮不选，然后明确继续默认路径。

开始前必须先输出：
1. 当前代码状态总结
2. 本轮选择的单一方向
3. 本轮准备连续完成的 `2` 到 `4` 个切片
4. 这些切片分别对应的现实场景和做完后的直接结果
5. 本轮明确不做的内容
6. 然后直接开始实现，不要停在计划阶段等确认

表达要求：
- 全程用通俗、自然、好懂的中文
- 先讲结论，再讲细节
- 少用英文术语，必要时顺手解释
- 不要写得像咨询报告
- 制定计划时，必须把“这次改代码对应的现实场景”讲清楚，不要只堆开发术语
- 尽量说成人能代入的结果，例如“这轮做完后，排障时不再只能翻本地命令，而是能直接从统一读面看到 worker 状态”

实现要求：
- 必须交付可运行代码
- 必须补必要的最小测试或最小验证
- 必须同步必要文档
- 如果环境缺依赖导致某些测试或命令无法运行，要明确说明缺什么、卡在哪里，并补充你实际完成的其他验证
- 同一条功能链路上的改动默认这一轮一并完成，不要再拆碎

文档要求：
- 若本轮影响项目对外状态、已实现能力、运行方式、已知限制，更新 `README.md`；不适合放首页的运行细节，优先下沉到合适文档
- 若本轮改变路线、关闭待办、拆分待办或调整优先级，更新 `doc/TODO.md`
- 若本轮改变文档入口、分工或索引关系，再更新 `doc/README.md`；否则不要为了留痕改它
- 每轮结束都要追加更新 `doc/history/memory-log.md`
- 所有文档都必须反映真实状态，不要预写未来能力
- 如果本轮未影响 `README.md`、`doc/TODO.md` 或 `doc/README.md`，必须明确说明为什么没改

完成后按这个顺序汇报：
1. 改了哪些文件
2. 落地了哪些能力
3. 哪些仍是 mock / stub / placeholder
4. 做了哪些测试或验证，哪些因环境限制没法做
5. 更新了哪些文档
6. `README.md` 是否更新；如果没更新，说明原因
7. `doc/TODO.md` 是否更新；如果没更新，说明原因
8. 下一步在同一方向下最合理的连续切片是什么

本轮目标：
<用一句能落到现实场景的话写清楚这轮目标，不要只写模块名>
```
