1. 本项目要构建的是一个公司化的 Agent Team 自动化开发框架。用户扮演董事会，只负责提出目标、约束和验收标准；Main Agent 扮演 CEO，接收董事会任务后，必须自主拆解、委派、推进并最终产出可交付物，而不是在常规阶段频繁等待用户反馈。

2. 框架默认采用“自治执行，少量审查门”的治理模式。除少数必须升级的节点外，Agent Team 应当自行完成需求澄清、方案设计、任务分解、编码实现、测试验证、缺陷修复与交付整理。系统应明确规定：如果不存在高风险阻塞条件，团队必须基于合理假设继续推进，不得把不必要的问题回抛给董事会。

3. 视觉与体验质量应被视为独立于代码质量的一级治理对象。当前 AI 在编码、语法、测试覆盖方面通常已能达到较高完成度，但在视觉审美、界面节奏、品牌感、一眼吸引力等方面仍明显不足，而用户满意度往往首先由视觉第一印象决定。因此，所有会形成用户可见视觉结果的关键里程碑，必须提交董事会审核。

4. 建议将董事会审核门严格限制在视觉交付物相关节点，包括但不限于：信息架构草图、页面视觉方向、关键界面初稿、设计系统定稿、高保真 UI、品牌主视觉、首页首屏、重要交互动效、最终视觉验收版。上述节点未获董事会确认前，可以继续做技术预研与实现准备，但不应擅自将视觉方向视为最终定稿。

5. 除视觉审核门之外，其他阶段原则上全部由团队自主闭环，包括：技术方案设计、模块拆分、接口定义、代码实现、单元测试、集成测试、回归验证、性能优化、日志与监控、文档整理、发布准备。董事会不参与日常执行细节，也不承担中间决策负担。

6. CEO Agent 必须具备持续推进责任，而不是仅做路由转发。其职责包括：定义阶段目标、给各子 Agent 分配清晰产出物、设定每阶段完成标准、检查是否达到进入下一阶段的条件、在成员停滞时重新分派任务、在交付偏离目标时强制返工，并确保项目始终朝着“产出可交付物”而不是“停留在讨论”前进。

7. 系统应内置明确的升级规则。只有在以下情况出现时，团队才允许向董事会请求决策：第一，视觉方向存在多个高成本备选路径，且会显著影响最终风格；第二，需求目标、预算、品牌定位或合规边界本身发生冲突；第三，执行所需权限、资源或外部依赖缺失，团队无法自行补足；第四，继续推进将导致明显返工风险。除此之外，默认继续执行。

8. 为了防止 GPT 类模型在执行中途习惯性等待反馈，框架应把“继续推进”设计成制度而不是建议。每个阶段都必须有明确输出格式，至少包含：阶段结论、关键假设、产生的 artifact、下一阶段输入、是否阻塞。如果不是正式阻塞态，Agent 不得输出“等待用户确认后继续”作为默认结束语。

9. 视觉审查应采用可比较、可投票、可追溯的交付形式，而不是模糊描述。每次提交给董事会的视觉里程碑，至少应包含：设计目标、适用场景、2 到 3 个候选方案、每个方案的风格说明、参考图或原型图、预期用户感知、风险提示、推荐方案。董事会只对方向作判断，不参与具体实现细节。

10. 最终目标不是做一个会不断请示的 Agent 系统，而是做一个“只有在关键审查门才升级，其余时间像公司一样自动运转”的交付系统：董事会定方向，CEO 保推进，团队做执行，视觉节点做审核，最终稳定地产生用户可验收的产品与资产。

11. 公司初始化时默认只存在一名 CEO Agent，不预设完整组织编制。CEO 在收到董事会任务后，应根据项目目标、复杂度、风险水平与预计交付周期，自主招聘一定数量的员工，逐步形成适合当前项目的最小健康团队规模，而不是固定生成一套同质化班底。

12. 公司的人才模型不能只以技能为中心定义。每一名员工在团队中的作用影响因子，必须由“技能”“性格”“审美”三类属性共同决定。技能决定能做什么，性格决定如何协作、如何判断风险与如何处理冲突，审美决定其对用户可见结果、表达质量和风格一致性的影响。

13. 公司必须显式追求同岗异质性，而不是同岗一致性。对于同一职位或相近职责的员工，如果其人格长期高度趋同，容易出现风险盲区、意见回声、错误放大、过度自信、审美单一或缺陷互相放过等问题。因此，CEO 在招聘时必须把“团队多样性”视为约束条件，而不是可选项。

14. 系统应为每名员工建立结构化人格画像，并在招聘与组队时计算相似度。人格画像至少可包含以下维度：风险偏好、质疑倾向、执行速度、严谨程度、抽象能力、实用主义倾向、沟通风格、用户同理心、冲突承受度、审美取向。对于同岗员工，不应允许人格向量高度重合到足以削弱团队鲁棒性的程度。

15. 同一岗位的候选员工不应只追求“最强”，而应追求“互补”。例如，一名偏激进、快速推进的工程员工，应搭配一名偏保守、强调验证和风险控制的员工；一名偏功能优先的设计员工，应搭配一名偏品牌与视觉一致性的员工。团队的整体质量来自互补性，而不是单点最优。

16. 核心员工的人格选择必须经过董事会批准。这里的核心员工，指会显著影响项目方向、产品体验、工程质量或组织风格的首批关键角色，例如技术负责人、视觉负责人、产品负责人、质量负责人等。CEO 可以提出候选名单和推荐理由，但最终应由董事会确认这些核心角色的人格组合是否符合公司治理目标。

17. 为降低董事会负担，董事会不需要审批所有员工，只审批“首批核心员工的人格与审美定位”。普通执行员工、补位员工和后续扩编人员，可由 CEO 在既定治理规则下自主招聘，但仍需满足团队异质性约束，并接受后续绩效与稳定性评估。

18. CEO 在提交核心员工候选时，应提供结构化招聘提案，而不是只报名字。每次提案至少应包含：候选角色、负责范围、技能优势、人格画像、审美画像、与现有成员的互补关系、潜在风险、推荐聘用理由、是否需要董事会审批。董事会审批的重点是团队组合是否健康，而不是具体 prompt 文案。

19. 公司应具备动态扩编与动态替换能力。随着项目进入不同阶段，CEO 可以新增、冻结、替换或淘汰员工，以保证团队在探索期、实现期、测试期、视觉打磨期和交付期都维持合适的人格与能力结构。组织应服务于项目，而不是要求项目去迁就固定组织。

20. 最终目标不是搭建一组“参数不同但本质相同”的 Agent，而是搭建一个具备真实组织韧性的公司：CEO 负责识别所需人格组合，董事会把关首批核心人格配置，团队通过技能互补、性格制衡和审美分层来提高决策质量、风险识别能力与最终交付质量。

21. 框架的消息传递与任务流转机制采用“事件溯源状态总线 + Ticket 驱动无状态执行器”架构，不采用群聊式多 Agent 长上下文对话作为主协作模式。系统的核心交互单位应是结构化事件、结构化状态和结构化工单，而不是自由文本聊天。

22. CEO 在运行时应被视为“计算无状态”的调度器，而不是“持久状态为空”的角色。CEO 自身不保留长对话记忆，每次仅在 Tick 触发时读取当前状态快照并输出标准动作；但公司的战略目标、审批结果、组织结构、预算、失败历史与产物索引必须被持久化到总线中，作为下一次调度的输入。

23. CEO 的唤醒必须由显式事件触发，典型触发源包括：董事会新指令、工单完成、工单失败、预算告警、熔断告警、审批结果返回、关键依赖满足、超时重试到期。CEO 不应被持续轮询式调用，也不应依赖“保持一段长对话”来维持上下文。

24. CEO 的输出必须被强制约束为标准动作，而不是自由文本建议。CEO 只能提出如 `CREATE_TICKET`、`RETRY_TICKET`、`REQUEST_BOARD_REVIEW`、`HIRE_EMPLOYEE`、`FREEZE_NODE`、`ESCALATE_INCIDENT`、`MARK_STAGE_DONE` 等受控动作；随后由确定性 Reducer 和 Transition Guard 校验动作是否合法，再提交到状态总线。

25. 所有 Agent 之间的正式信息交接必须通过总线完成，禁止未建模的黑箱式直接对话。系统不禁止协作本身，但要求所有协作都必须转化为可审计的事件、工单、产物引用或审批记录，确保可重放、可追踪、可调试。

26. 所有员工 Agent 均为无状态 Ticket 执行器。员工只处理当前单一节点的原子任务，不继承隐式聊天历史，不依赖全局长上下文，不保留私有会话记忆。员工的输入只能来自 CEO 或调度器生成的标准工单，输出只能是符合 schema 的结构化结果与产物引用。

27. 标准工单必须至少包含以下字段：`ticket_id`、`workflow_id`、`node_id`、`parent_ticket_id`、`attempt_no`、`role_profile_ref`、`constraints_ref`、`input_artifact_refs`、`context_query_plan`、`acceptance_criteria`、`output_schema_ref`、`output_schema_version`、`allowed_tools`、`allowed_write_set`、`retry_budget`、`priority`、`timeout_sla`、`deadline_at`、`escalation_policy`、`idempotency_key`。缺失这些控制字段的 prompt 不应被视为有效工单。

28. 工单中的上下文生成应由独立的 Context Compiler 负责，而不应完全依赖 CEO 手工筛选文本。CEO 决定“需要哪些证据与引用”，Context Compiler 根据工单声明、状态树、产物索引、检索规则和约束策略，生成最小必要上下文。这样可以避免 CEO 成为信息瓶颈，并降低上下文裁剪失误导致的幻觉风险。

29. 底层控制面应采用 SQLite 作为嵌入式状态总线，负责存储事件日志、状态投影、工单池、租约、审批、预算和索引信息；大体积产物、代码快照和报告文件应存入文件系统或对象目录，再由 SQLite 保存引用。SQLite 负责控制面，不负责承载所有大对象正文。

30. SQLite 并发控制不能被视为天然无问题。系统必须显式采用 `WAL` 模式、`busy_timeout`、乐观版本号、Ticket 租约和幂等键策略，保证单机高可靠运行；任何涉及并发领取、重复执行、状态覆盖和超时回收的逻辑，都必须通过状态机与租约协议解决，而不是依赖数据库默认行为。

31. 系统应采用“事件日志 + 状态投影”双层模型。事件日志负责只追加写入，记录所有真实发生的动作；状态投影负责为 CEO、Board Gate、执行器和监控组件提供当前快照视图。CEO 读取的是投影状态，而不是原始聊天记录。

32. 为了支持企业规范、历史复盘与厚文档检索，系统可以在控制面之外挂接轻量检索层。第一层可使用 SQLite FTS5 处理关键词检索；第二层可使用 LanceDB 承载语义检索与长文档向量索引。RAG 能力应通过工单中的 `context_query_plan` 进入执行链路，而不是让 Agent 自由漫游历史记录。

33. 本消息机制的目标是显著降低幻觉概率，而不是宣称“绝对抗幻觉”。真正的抗幻觉能力来自受控输入、受控动作、确定性 Reducer、Schema 校验、工具结果验证、写集限制和状态机守卫，而不是来自单一的总线设计口号。

34. 当工单出现重复失败、同一错误指纹多次重现、测试连续回滚、写集冲突、预算异常燃烧或依赖长期无法收敛时，系统必须触发熔断。熔断事件应携带错误快照、失败计数、相关 Ticket、相关产物与推荐处置动作，向 CEO 或董事会升级，而不是让 Token 在隐性循环中持续消耗。

35. 董事会审批门也必须纳入同一消息机制。视觉里程碑、核心员工人格审批、预算超限决策等需要人类确认的节点，应统一建模为 `BOARD_REVIEW_REQUIRED`、`BOARD_REVIEW_APPROVED`、`BOARD_REVIEW_REJECTED` 等事件，并通过状态机冻结或解冻相关下游节点，禁止 CEO 绕过审批门直接推进最终交付。

36. 事件模型需要进一步区分“领取任务”和“开始执行”，因此正式事件集中应至少包含：`SYSTEM_INITIALIZED`、`WORKFLOW_CREATED`、`WORKFLOW_COMPLETED`、`WORKFLOW_ESCALATED`、`TICKET_CREATED`、`TICKET_LEASED`、`TICKET_STARTED`、`TICKET_COMPLETED`、`TICKET_FAILED`、`TICKET_TIMED_OUT`、`TICKET_RETRY_SCHEDULED`、`TICKET_CANCELLED`、`ARTIFACT_WRITTEN`、`BOARD_REVIEW_REQUIRED`、`BOARD_REVIEW_APPROVED`、`BOARD_REVIEW_REJECTED`、`CIRCUIT_BREAKER_OPENED`、`CIRCUIT_BREAKER_CLOSED` 等类型，避免语义混叠。

37. Reducer 必须保持纯净。LLM 只能提出动作，Reducer 只能校验动作是否合法并给出接受结果与派生效果，真正的事件写入必须由运行时提交完成。系统不得让 Reducer 成为隐藏副作用组件。

38. `TICKET_COMPLETED` 的验收必须拆分为两类校验：第一类是 `schema validation`，核对返回结果是否符合 `output_schema_ref` 与版本要求；第二类是 `write-set validation`，核对 Worker 是否只写入了 `allowed_write_set` 允许的目标。两类校验任一失败，都必须转为受控失败流。

39. 发给 Worker 的最终输入应是“编译后的执行包”，而不只是控制面上的 Ticket 引用。控制面可以保留 `role_profile_ref` 与 `constraints_ref` 以便审计，但实际执行时必须展开成 `compiled_role`、`compiled_constraints` 和 `atomic_context_bundle`，确保 Worker 拿到的输入是原子、封闭且可直接执行的。

40. 视觉审批状态机应采用 `PENDING -> EXECUTING -> BLOCKED_FOR_BOARD_REVIEW -> REWORK_REQUIRED -> COMPLETED` 的显式模型。董事会打回后不得简单回退为泛化的 `PENDING`；系统应保留“已审查且被拒绝”的事实，并且只冻结依赖该视觉节点的下游子图，而不是冻结整个工作流。

41. 在标准 Ticket 流之外，系统可以提供一种受控例外机制：会议室协议（Meeting Room Protocol），用于处理高度耦合、跨角色冲突明显、串行返工难以收敛的议题，例如前后端接口定稿、交互与实现成本冲突、架构与测试策略冲突等。

42. 会议室协议不得退化为自由群聊。它必须被建模为临时子工作流或特殊协同节点，只能在明确触发条件下开启，并受固定参与者、固定议题、固定输入材料、固定 Token 预算、固定轮次上限和固定输出 schema 约束。

43. 会议室协议应采用显式事件，例如 `MEETING_ROOM_REQUESTED`、`MEETING_ROOM_OPENED`、`MEETING_ROUND_COMPLETED`、`MEETING_CONSENSUS_SUBMITTED`、`MEETING_NO_CONSENSUS`、`MEETING_ROOM_CLOSED`。会议过程仍然必须受总线治理，不允许在总线之外形成不可审计的黑箱讨论。

44. 会议室的目标不是保留完整聊天记录，而是产出结构化共识。会议结束时，必须由指定书记员或总结器输出 `Consensus_Document`，内容至少包含：议题、参与者、输入物引用、已达成共识、被否决方案、未决分歧、接口约定、风险、后续工单和责任人。系统可以丢弃完整 transcript，但至少应保留摘要、哈希或压缩审计线索。

45. 会议室协议应优先采用结构化轮次而不是开放式争论，至少可拆成：立场轮、质疑轮、方案轮、收敛轮、闭会轮。会议超出预算、轮次或时间上限后必须结束；若仍无法收敛，应触发 `MEETING_NO_CONSENSUS` 并升级给 CEO 或董事会，而不是无限拉长讨论。

46. 系统应引入 `Maker-Checker` 同行评审机制作为默认质量控制层之一。重要产物不得由执行者（Maker）直接提交 CEO 或直接进入主干；在满足合入条件前，必须先经过独立审查者（Checker）的内部评审，以降低 CEO 的逐件验收负担，并减少单个 Agent 幻觉、疏漏或过度自信导致的质量风险。

47. Checker 不应被设计成情绪化“怒骂型”人格，而应被设计为对抗性、怀疑性、证据导向的审查角色。其职责是主动寻找缺陷、矛盾、边界遗漏、规范偏离、风险低估与实现漏洞，并基于证据提出审查结论；系统追求的是高质量质控，而不是高噪音内耗。

48. 当 Maker 完成代码、方案、文档、测试或其他关键产物后，运行时应自动创建对应的 `REVIEW_TICKET`，并优先派发给与 Maker 在风险偏好、思维风格或审美取向上具有互补性的 Checker。Maker 与 Checker 不应是职责混同的同质副本，否则评审将退化为形式化背书。

49. Checker 的输出必须是结构化内审裁决，而不是自由发挥的主观评论。审查结果至少应包含：问题标识、严重级别、问题类别、证据引用、定位信息、要求动作、是否阻塞等字段；审查结论至少应支持 `APPROVED`、`APPROVED_WITH_NOTES`、`CHANGES_REQUIRED`、`ESCALATED` 四类状态，以区分轻微意见、必须返工与超出 Checker 裁量边界的情况。

50. 只有获得 Checker 的可接受裁决后，产物才可进入主干、进入下游关键节点或提交给更高治理层；若结果为 `CHANGES_REQUIRED`，系统应将结构化 findings 编译为新的 `FIX_TICKET` 返回给 Maker 修正，而不是让 CEO 手工转述审查意见。若相同问题指纹在 Maker-Checker 闭环中重复出现达到阈值，则必须触发熔断、升级或换人，防止质量机制本身演化为无穷返工回路。

51. 框架应提供一个极简、高密度、偏黑客控制台气质的 `Boardroom UI` 作为人机治理控制面，而不是聊天窗口、办公桌宠物动画或泛化的项目管理后台。其职责是展示状态投影、审批请求、执行器状态、事件心跳和系统健康，而不是承载工作流核心逻辑。

52. `Boardroom UI` 应坚持 projection-first 原则。主界面主要消费后端投影状态而不是原始 prompt 或长聊天记录；前端可以持有筛选、展开、比较、审查草稿等界面级临时状态，但不得拥有工作流真相、调度规则、预算守卫或审批合法性判断。

53. 主界面的默认结构应收敛为“审批收件箱 + 流水线主视图 + 执行器与事件流 + 常驻健康条”的控制面布局。DAG 依赖图不应作为首页主舞台，但必须保留为二级依赖检查器，用于排查冻结、死锁、阻塞链和关键路径，而不是被完全删除。

54. 董事会审批不应以小模态框承载，而应通过 `Inbox -> Review Room` 进入专门的审查界面。董事会默认看到的应是面向裁决的 `Board Review Pack`，包含审批对象、触发原因、产物预览、关键证据摘要、与上一版差异、Maker-Checker 结论、风险与预算影响；原始 `CompiledContextBundle` 只应作为高级调试视图存在。

55. 前后端通信应采用稳定的 `snapshot + stream + command` 模式：前端通过 projection API 拉取快照，通过事件流订阅增量变化，并通过命令接口提交审批、初始化和配置动作。MVP 阶段优先采用 SSE 作为单向事件同步通道，并显式支持 cursor 恢复、重连、心跳和全量重同步。

56. 模型与 Provider 配置中心应被视为系统级能力，而不是简单表单。除 `base_url`、`api_key`、模型列表和参数外，还应支持能力标签、岗位绑定、任务级覆盖、fallback 路由、健康检查、并发与限流、成本等级、延迟等级、上下文窗口和结构化输出能力等元数据。

57. BYOK 方案应优先服务跨平台部署与普通用户可用性，而不是引入复杂的系统级密钥基础设施。系统可以允许用户在 UI 中直接配置并本地保存 API Key，并在保存后进行遮罩展示；同时提供环境变量导入、手动轮换和清晰的本地信任边界提示。该项目属于本地或自托管便利型产品，可以接受“安全性适度让位于可移植性与低门槛”的设计取舍，但必须明确告知这种权衡。

58. 框架应支持角色级默认模型绑定、任务级临时模型覆盖，以及 `preferred model` 与 `actual model` 的双记录。系统既要表达“本来希望谁来做、用什么模型做”，也要表达“最后实际走了哪个 Provider / 模型 / fallback 路径”，保证调度、审计和复盘都有依据。

59. 模型路由不能只看可用性，还应显式支持成本等级与参与频率策略。高价模型应允许只用于少量高杠杆、低频参与的角色或任务，而不是默认让全体角色长期绑定同一高成本模型。

60. 公司治理模板应支持定义低频、高成本、以文档产出为主的核心角色，例如 CTO 或架构师。这类角色可以参与关键方案与治理决策，但默认不承担日常编码、测试和持续实施主力工作。

61. CTO / 架构师类角色的默认产物应是治理和设计文档，而不是直接代码实现。至少应覆盖：架构方案、技术选型、里程碑拆解、详细设计、TODO 或 backlog 建议，并且这些文档应能被 CEO 继续编译成后续执行工单。

62. CEO、文档型角色、实施 Worker 的模型选择应彼此解耦。治理模板可以给出默认建议组合，例如 CEO 使用常用主力模型、CTO / 架构师使用高成本低频模型、实施 Worker 使用常规编码模型，但框架层不应把某一家模型厂商硬编码成唯一合法方案。
