# 图书馆管理系统独立演练设计

## 状态

- 状态：已归档，不再作为后续实现依据
- 日期：2026-04-08
- 范围：独立演练，不接入主线 `workflow/ticket`

## 1. 目标

这次演练要验证一条比“脑筋急转弯并行答题”更复杂的协作链：

- CEO 能基于现有组织架构发起一轮真实任务分派
- 组织内至少形成一条有效协作链：有人实施、有人测试、有人运维启动
- 最终产出一个可本地快速启动的“图书馆管理系统”最小成品
- 整个过程留下可供董事会审核的中文材料，且可读性高，不写成技术流水账

这次演练关注的是流程闭环和审计闭环，不追求完整产品化。

## 2. 边界

### 2.1 这是独立演练，不是主线能力扩张

- 不把这次演练直接接进现有主线 `workflow/ticket`
- 不要求所有员工都被派发任务
- 不要求本轮未被使用的员工参与补位或凑数
- 不把这次流程文件登记为过程资产

### 2.2 这次必须真实做到的事

- CEO 生成中文任务简报
- CEO 生成中文分工台账，并记录谁被派发、谁未被派发、原因是什么
- 至少一名实施角色完成交付
- 至少一名测试角色完成检查记录
- `platform_sre` 拉起最终成品，供董事会审核
- 生成项目全生命周期简报，面向董事会，用高可读性中文说明全程

### 2.3 这次明确不做的事

- 复杂登录鉴权
- 真实支付、短信、邮件通知
- 复杂报表、权限中心、多租户能力
- 与现有前端主线深度集成

## 3. 最小成品定义

本轮“图书馆管理系统”收口成一个可快速启动的最小毕业设计成品，至少包含：

- 图书列表
- 图书搜索
- 借阅登记
- 归还登记

为了保证演练能稳定落地，这个成品可以采用“最小全栈”结构：

- 前端：单页界面，展示图书列表和操作入口
- 后端：最小接口服务，返回图书和借阅状态
- 数据层：内存数据或轻量本地数据文件，包含演示种子数据
- 启动方式：一条可执行命令或一个脚本，能快速拉起演示环境

董事会看到的重点不是工程复杂度，而是“这个系统确实能打开、能演示、能解释”。

## 4. 组织分工

这次直接复用现有组织结构，不额外补招。

- CEO：生成任务简报、分工台账、阶段汇总、最终董事会报告
- `governance_architect`：产出系统边界、模块拆分、接口约定、数据流说明
- `governance_cto`：产出任务切片、交付顺序、验收口径、风险提示
- `backend_engineer`：实现后端接口与借阅归还逻辑
- `database_engineer`：提供数据结构、种子数据、状态约束
- `frontend_engineer`：实现最小界面和基本操作流
- `checker`：执行检查并给出中文验收结论
- `platform_sre`：负责启动、运行说明、启动结果记录

允许本轮存在“未被派发员工”。但 CEO 的分工台账必须显式记录：

- 本轮被派发了哪些人
- 每个人拿到了什么任务
- 哪些人未被派发
- 未派发的原因

## 5. 执行流程

建议把这次演练固定成六个步骤：

1. CEO 生成中文任务简报。
2. CEO 基于现有组织生成分工台账，并向被派发员工下发中文上下文。
3. 被派发员工并行交付：架构、计划、实现、测试、运维准备各自产出。
4. CEO 汇总交付结果，必要时进行一轮补充分派。
5. `platform_sre` 启动图书馆管理系统，记录启动证据。
6. CEO 生成董事会审核包和项目全生命周期简报。

这条链的关键是“CEO 能派、有人接、有人交、有人测、有人拉起、董事会能看”。

## 6. 归档策略

### 6.1 归档文件

下面这些材料都要保留，但统一按“归档文件”处理，不登记为过程资产：

- CEO 任务简报
- CEO 分工台账
- 已派发员工的上下文
- 各角色交付摘要
- 测试记录
- 运维启动记录
- 项目全生命周期简报
- 最终董事会报告

归档文件需要满足两个要求：

- 使用中文
- 能单独阅读，不依赖代码上下文也能理解

### 6.2 董事会审核材料

董事会最终至少看到五组内容：

1. 任务简报：这次做什么、为什么这样收口。
2. 分工台账：CEO 把什么任务派给了谁，哪些人没参与。
3. 完成情况：每个人交了什么，是否完成。
4. 启动证据：运维如何启动，结果如何，董事会看什么入口。
5. 全生命周期简报：用高可读性中文总结项目从立项到交付的全过程。

## 7. 产物结构

建议这次演练至少产出两类文件：

### 7.1 成品交付物

- 图书馆管理系统前端
- 图书馆管理系统后端
- 本地数据或种子数据
- 启动脚本或启动命令

### 7.2 审计与汇报文件

- `brief`：CEO 任务简报
- `assignment ledger`：分工台账
- `dispatch contexts`：员工派发上下文
- `execution summaries`：角色执行结果摘要
- `test report`：检查结果
- `launch record`：运维启动记录
- `lifecycle brief`：项目全生命周期简报
- `board report`：最终董事会报告

## 8. 状态判定

### `COMPLETED`

同时满足下面条件：

- CEO 完成至少一轮有效派发
- 至少有实施角色交付内容
- 至少有测试角色提交检查结果
- `platform_sre` 成功拉起成品
- 审计与汇报文件齐全

### `COMPLETED_WITH_GAPS`

系统可以启动，但存在下面任一情况：

- 部分任务未完成
- 部分角色未交付
- 功能有明确缺口，但不影响最小演示
- 汇报中记录了待补问题

### `FAILED`

出现下面任一情况：

- CEO 没有形成有效派发链
- 没有形成“实施 + 测试 + 启动”的最小闭环
- 成品无法启动
- 审计材料缺失到无法复盘

## 9. 错误处理

这次演练应保留“失败也归档”的原则。

即使最终状态为 `FAILED`，也仍然要生成：

- 当前已完成的分工记录
- 已派发上下文
- 已收到的角色结果
- 启动失败记录
- 面向董事会的失败说明

这样董事会才能看到“流程哪里断了”，而不是只看到一个空结果。

## 10. 测试与验证口径

这次验证按“最小有效证据”收口，只检查四件事：

1. 管理材料齐全：任务简报、分工台账、全生命周期简报、董事会报告都存在。
2. 执行链存在：至少有实施角色、测试角色、运维角色三类有效结果。
3. 成品可演示：能展示图书列表，并完成搜索、借阅、归还中的最小演示。
4. 启动可复核：运维记录了启动命令、启动状态和访问入口或页面证据。

这里不要求完整自动化测试矩阵，也不要求每位员工都真正参与。

## 11. 实现建议

为了尽快落地，建议沿用现有独立演练模式，新增一个与“脑筋急转弯演练”并行的独立场景：

- 场景负责生成分工、并行执行、汇总结果、归档材料
- 场景内部产出一个最小图书馆管理系统成品
- 运维步骤在场景尾部执行，并把启动结果纳入董事会报告

这样可以最大限度复用已有 provider、角色目录、归档写入和董事会报告结构，同时避免碰现有主线和前端历史问题。
