# 图书馆管理系统主线自动推进设计

## 状态

- 状态：已确认，待实现
- 日期：2026-04-08
- 范围：当前图书馆项目实例生效，走真实主线 `workflow/ticket`

## 1. 目标

这次要验证的不是独立演练脚本，而是框架主线本身：

- 用真实 `project-init -> scope review -> BUILD -> CHECK -> REVIEW -> closeout` 主链推进一个图书馆管理系统项目
- 本次项目允许 CEO 代替董事会在 board-facing 停点上做裁决，但只对当前项目实例生效
- 用户只需给出模糊北极星，CEO 负责先做需求征询和需求打包，再交给架构师和 CTO 切分
- 任务拆分必须足够细，形成大量可审计的原子任务，而不是“前端一次性产出前端、后端一次性产出后端”
- 允许出现真实的环状流转，例如 `实施 -> 检查 -> 返工 -> 再检查`
- 最终要产出一个精美、可用、可展示的图书馆管理系统，并生成高可读性的全链路报告

## 2. 旧材料处理

本轮开始前，之前的独立演练材料统一按“已归档，不复用”处理：

- 旧独立演练产物仍保留在归档区
- 旧独立演练设计文档不再作为实现依据
- 当前主线项目不读取旧演练归档作为输入
- 新项目从新的 workflow 开始，使用新的审计链

## 3. 项目级 CEO 代审授权

### 3.1 授权范围

这次新增的是**当前项目级别**的 CEO 代审授权，不是全局开关。

授权只对当前图书馆项目的 workflow 生效，至少覆盖：

- 初始化需求澄清板审
- `scope review`
- 最终 `REVIEW`
- 运行过程中真正升级到董事会的 board-facing incident 或改约束停点

### 3.2 不改变的内部质量门

下面这些节点仍保持现有内部治理规则，不由 CEO 代替：

- `BUILD` 内部 maker-checker
- `CHECK` 里的 checker 审查
- `closeout` 内部收尾审查

这次的目标是“董事会不再成为人工卡点”，不是“删除框架内部质量链”。

### 3.3 审计要求

每一次 CEO 代审都必须留下结构化记录：

- 原本该由董事会裁决的 review pack 是什么
- 当前 workflow 为什么允许 CEO 代审
- CEO 选了哪个选项
- 决策理由是什么
- 决策后自动生成了哪些 follow-up ticket

## 4. 入口与需求收敛链

### 4.1 初级用户输入假设

后续这条能力面向初级用户，不能假设用户一开始就能给出清晰需求。

用户可能只会给一句模糊北极星，例如：

- 做一个图书馆管理系统毕业设计

### 4.2 CEO 的前置职责

所以主线入口不能直接把模糊北极星交给架构师切分。必须先经过 CEO：

1. 用户提交模糊北极星。
2. CEO 发起一轮需求征询，澄清目标、边界、演示重点、最低验收、时间和复杂度取舍。
3. CEO 把征询结果整理成一份更具体的项目简报。
4. 架构师消费这份简报做系统拆分。
5. CTO 再把拆分结果压成大量原子任务。

### 4.3 和现有主线的关系

这条需求收敛链应尽量复用现有主线能力：

- 能命中 `force_requirement_elicitation` 或现有 requirement elicitation 路径时，优先复用主线现成入口
- CEO 生成的项目简报应作为后续治理票和实施票的上游输入
- 架构师和 CTO 不应直接面对“模糊北极星原文”，而应消费 CEO 整理后的项目简报

## 5. 真实主链推进方式

### 5.1 总体链路

这次项目按真实主链推进：

1. `project-init`
2. 需求征询 / 需求澄清
3. `scope review`
4. `BUILD`
5. `CHECK`
6. 最终 `REVIEW`
7. `closeout`

差别只在于：本次项目里的 board-facing 审批节点由 CEO 自动代审，而不是等待人工董事会。

### 5.2 自动推进原则

- 审批节点仍然存在，不会被物理跳过
- review pack 仍然会生成
- 审批记录、审计事件、follow-up 生成逻辑仍然保留
- 系统只是在发现“当前 workflow 允许 CEO 代审”时，由 CEO 自动提交裁决

## 6. 原子任务拆分原则

### 6.1 不允许粗粒度分派

本次实施层不能出现这类粗票：

- “后端把所有后端代码一次做完”
- “前端把所有前端代码一次做完”
- “数据库把所有数据层一次做完”

### 6.2 必须细粒度拆票

每张原子票只做一个很窄的结果面，示例包括：

- 定义图书领域模型
- 定义借阅状态流转
- 定义搜索接口约定
- 实现图书列表接口
- 实现借阅接口
- 实现归还接口
- 准备种子数据
- 实现图书列表页面
- 实现搜索框与筛选
- 实现借阅前端交互
- 实现归还前端交互
- 实现启动脚本
- 执行冒烟测试
- 执行视觉润色
- 整理最终演示包

### 6.3 数量目标

为了让主线真实体现“大型工程的分步流转能力”，本次项目应至少拆到 **20 到 40 张原子票**，优先保证：

- 每张票都能独立说明目的
- 每张票都能独立审查
- 多张票之间能形成明确依赖
- 原子票总量足够支撑最终审计展示

## 7. 环状流转要求

这次主线不应被设计成单次线性推进。必须允许并记录真实回流：

- 实施票完成后进入 checker
- checker 不通过时生成 fix ticket
- fix ticket 完成后重新进入 checker
- 平台启动失败时生成修复票，再次启动
- 最终 review 发现展示或交付整理不足时，补发细粒度返工票

最终报告必须能明确说明：

- 哪些票一次通过
- 哪些票经过返工
- 哪些票形成了小闭环

## 8. 治理层与实施层分工

### 8.1 治理层

CEO、架构师、CTO 负责把项目收清楚并切出任务：

- CEO：需求征询、项目简报、代审决策、阶段总结
- 架构师：系统边界、模块职责、接口与数据流
- CTO：任务切片、依赖收口、交付顺序、风险口径

### 8.2 实施层

实施票要真正进入现有主线支持的 runtime path：

- `frontend_engineer`
- `backend_engineer`
- `database_engineer`
- `platform_sre`
- `checker`

### 8.3 边界适配

当前主线边界要求：

- `backend / database / platform` 已进入正式 `BUILD` follow-up path
- 但它们仍不进入 direct CEO create-ticket

所以这次方案必须遵守现状：

- CEO 先创建治理层票和 review 所需票
- 通过治理结果和 scope follow-up 去生成大量细粒度实施票
- 不去硬改现有“CEO 直接给 backend/database/platform 建实施票”的边界

## 9. 测试纳入主线

这次不再把测试放在旁路脚本里。测试要成为主线正式阶段的一部分。

### 9.1 项目测试

- `CHECK` 阶段由 `checker` 真实执行
- checker 消费前面实施阶段的交付物
- checker 输出结构化检查结果
- 不通过时允许继续回流到细粒度修复票

### 9.2 框架测试

为了保证这套主线能力可回归，还要补自动化验证：

- 当前项目级 CEO 代审只作用于当前 workflow，不污染全局
- 从 `project-init` 到 `closeout` 能自动推进到完成
- 原子票数量、依赖链和返工链都有可验证输出
- 最终能产出面向人的工作流总报告

## 10. 最终作品定义

最终交付不是独立演练壳，而是一个通过真实主链推进出来的图书馆管理系统。

最小可用作品至少包含：

- 图书列表
- 图书搜索
- 借阅登记
- 归还登记
- 可本地启动的运行方式

同时，这次作品要求比上一轮更高：

- 页面要有明显完成度
- 可演示，不只是接口存在
- 交互和展示至少达到“毕业设计可展示”的水平

## 11. 最终报告要求

最后提交给用户和董事会的报告必须是**高可读性的中文组织形式**，不能写成事件流水账。

建议固定包含六部分：

1. 项目概览  
   说明项目目标、授权范围和最终结果。

2. 授权说明  
   说明哪些本该由董事会裁决的节点，本次改由 CEO 代审。

3. 工作流总览  
   按阶段展示 `project-init -> scope review -> BUILD -> CHECK -> REVIEW -> closeout`。

4. 原子任务分派表  
   逐张列出 ticket：谁接的、依赖什么、输出什么、当前状态、是否返工。

5. 实施链条  
   说明治理票如何生成实施票，实施票如何进入检查，哪些票被打回，如何再次推进。

6. 作品与结论  
   说明最终作品有哪些能力、测试结果如何、启动证据是什么、还剩哪些缺口。

## 12. 状态判定

### `COMPLETED`

同时满足下面条件：

- 当前项目主链自动推进到 `closeout`
- CEO 代审链真实生效
- 形成足够细的原子票链
- 测试真实进入 `CHECK`
- 最终作品可启动、可展示
- 报告能清楚展示原子任务和实施链条

### `COMPLETED_WITH_GAPS`

系统能跑完主线，但存在下面任一情况：

- 原子票拆分还不够细
- 最终作品有明显缺口但仍可演示
- 某些返工链条未完全收尾
- 报告结构已成形，但仍有部分信息不完整

### `FAILED`

出现下面任一情况：

- 当前 workflow 没有形成自动推进闭环
- CEO 代审没有真正接管 board-facing 停点
- 原子票链没有建立起来
- 主线卡死在 review 或 closeout 前
- 最终作品无法启动

## 13. 实现建议

为了尽快落地，这次实现建议按四块收口：

1. 新增当前项目级 CEO 代审授权模型与自动裁决路径。
2. 把需求征询和项目简报收进主线入口前半段。
3. 增强治理票到实施票的任务细粒度拆分能力，确保产生大量原子 ticket 和可回流链。
4. 新增主线级全链路报告生成能力，用高可读性中文输出整条任务分派与实施链。

这四块都属于当前项目实例内的主线能力增强，不改变全局默认董事会治理规则。
